# Project Overview

## Project Name

rakugaking

## One-Line Concept

A chaotic 3D drawing battle game where players draw characters, assign simple commands, and watch them fight automatically.

## Core Tagline

Draw, build, program, and laugh.

## Project Idea

Players draw a character by body parts, such as head, body, upper arms, arms, hands, thighs, shins, and feet.

The drawn parts become a physics-based character. The player assigns simple behavior commands to parts, then the character fights another drawn character automatically.

The game is intentionally messy. Characters may fall, miss attacks, spin around, or move in unexpected ways. That chaos is the main fun.

## Core Experience

### 1. Draw

The player draws body parts. The shape of each part affects how the character moves.

Examples:

- Long arms may reach farther but become unstable.
- Short arms may be stable but fail to reach the opponent.
- Large heads may make the character easier to topple.
- Thick legs may improve stability but reduce agility.

### 2. Program

The player does not write real code.

Programming means choosing simple repeated actions, such as:

- punch
- swing arm
- guard
- kick
- jump
- crouch

The command system must be easy enough for children to understand.

### 3. Battle

After setup, the player does not directly control the character.

The character automatically fights according to its drawing, body structure, simple behavior commands, and physics simulation.

## Main Differentiator

The drawing itself becomes the strategy.

The same command can behave differently depending on how the player drew the character.

## Character Direction

Future character forms may include:

- humanoid
- dog-like
- bird-like
- lizard-like
- giant
- yokai
- monster

However, the command system should remain mostly shared across character types.

## Target Users

- Children
- Adults
- Casual players
- Creative players
- Players who enjoy funny physics games

## Target Platforms

- Smartphone
- PC

## Technology Stack

- Unity

## Online Direction

Online battle and character sharing are important product goals and should remain in the plan.

The MVP should consider these from the beginning, even if implementation is staged.

## Monetization Direction

Monetization should be considered from the beginning.

Possible monetization areas:

- additional drawing tools
- cosmetic effects
- character slots
- skeleton/body type packs
- decoration parts
- online sharing features

Pay-to-win design should be avoided.

## Product Principle

The game should not become a difficult programming tool.

It should feel like a toy, a drawing tool, and a silly battle game combined.
